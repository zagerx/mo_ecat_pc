# mo_ecat_pc 工程框架说明

> 文档目的：说明 `mo_ecat_pc` 与 `mo_ecat_core` 的职责划分、接口契约和边界条件，作为后续 UI 开发和核心库对接的总纲。
> 更新日期：2026-06-25

## 1. 项目定位

`mo_ecat_pc` 是 EtherCAT 主站上位机工程，当前主要工作是开发 Qt6 UI。它负责把 EtherCAT 主站能力呈现给用户，并提供扫描、配置、运行、诊断、日志和状态监控等人机交互功能。

`mo_ecat_core` 是 EtherCAT 主站核心库。它负责网卡初始化、从站扫描、主站状态机、PDO/SDO 通信、从站状态检查、日志输出和后续运动控制相关能力。核心库必须保持纯 C++，不依赖 Qt，也不依赖 UI 工程的任何类型。

两者之间的关系应当是：

```text
Qt UI / ViewModel
        |
        | Qt signal/slot, Qt model, UI command
        v
core_bridge
        |
        | mo_ecat_core public C++ API
        v
mo_ecat_core
        |
        | SOEM / raw Ethernet
        v
EtherCAT slaves
```

## 2. 职责划分

| 模块 | 职责 | 不负责 |
|------|------|--------|
| `mo_ecat_pc` | Qt6 UI、窗口、表格、日志显示、按钮命令、配置输入、用户工作流 | EtherCAT 协议细节、SOEM 直接调用、从站状态机实现 |
| `core_bridge` | 持有并调度 `MoEcatMaster`，把 core 回调转换为 Qt 信号/数据模型，把 UI 命令转换为 core 调用 | 直接操作 UI 控件、实现 EtherCAT 协议、绕过公共 API 访问 core 内部 |
| `mo_ecat_core` | EtherCAT 主站核心能力、公共 API、状态机、SDO/PDO、日志、从站信息 | Qt 类型、界面刷新、用户交互、窗口线程管理 |
| `SOEM` | EtherCAT 底层通信库 | 上位机业务逻辑、UI、应用状态编排 |

## 3. 依赖方向

依赖方向必须单向：

```text
mo_ecat_pc/app
  -> core_bridge
    -> mo_ecat_core/include/mo_ecat
      -> mo_ecat_core/src
        -> SOEM
```

禁止反向依赖：

- `mo_ecat_core` 不得 include Qt 头文件。
- `mo_ecat_core` 不得依赖 `mo_ecat_pc` 的 UI、bridge、model 或配置类型。
- UI 层不直接访问 `mo_ecat_core/src` 下的内部头文件。
- UI 层不直接调用 SOEM。
- 如需扩展能力，优先通过 `include/mo_ecat/` 增加公共接口，而不是让 UI 依赖内部实现。

## 4. 接口边界

当前 `mo_ecat_core` 对上位机暴露的入口是 `mo_ecat::MoEcatMaster`，公共头文件位于：

```text
mo_ecat_core/include/mo_ecat/
├── master.h
├── config.h
└── types.h
```

接口分为四类：

| 类别 | 典型接口 | UI 使用方式 |
|------|----------|-------------|
| 生命周期 | `InitializeAdapter`、`Scan`、`EnterMaintenance`、`PrepareRun`、`StartOperation`、`BackToMaintenance`、`Stop` | 由 UI 命令触发，经 `core_bridge` 排队到 core 所在线程执行 |
| 状态查询 | `GetState`、`GetSlaveCount`、`GetSlaveInfos`、`GetSnapshot` | bridge 拉取后转换为 Qt model 或状态对象 |
| 诊断调试 | `ReadSdo`、`WriteSdo`、`DumpPdoMapping` | 视为可能阻塞操作，不在 UI 主线程直接调用 |
| 回调通知 | `on_state_changed`、`on_log_message`、`on_fault`、`on_feedback` | bridge 接收后用 queued signal 或线程安全队列发给 UI |

详细说明见：

- [mo_ecat_core 公共接口说明](接口边界/mo_ecat_core公共接口.md)
- [UI 与 core_bridge 边界说明](接口边界/UI与core_bridge边界.md)

## 5. 线程模型原则

推荐模型：

```text
Qt GUI thread
    |
    | queued signal/slot
    v
CoreWorker thread
    |
    | owns MoEcatMaster
    | periodically calls Service()
    v
mo_ecat_core
```

原则如下：

1. `MoEcatMaster` 实例由 `core_bridge` 的 worker 线程持有。
2. `Service()` 由 worker 线程周期调用，不由 UI 主线程直接调用。
3. UI 按钮命令通过 Qt queued connection 投递给 worker。
4. SDO 读写、扫描、状态切换等可能耗时的调用不在 UI 主线程执行。
5. core 回调不能直接操作 UI 控件，必须转换为 Qt queued signal 或写入线程安全队列。

当前 core 实现中需要特别注意：

- `on_state_changed` 在 `Service()` 调用线程中触发。
- `on_log_message` 由 core 内部异步 Logger 的后台线程触发。
- `on_fault` 和 `on_feedback` 当前在公共头文件中定义，但实现层尚未实际触发，应作为待补齐能力处理。

## 6. 数据边界

core 对外只输出普通 C++ 数据结构，例如：

- `MasterState`
- `SlaveInfo`
- `SlaveFeedback`
- `MasterSnapshot`
- `EcMasterConfig`

UI 不直接持有 core 内部对象。bridge 负责把这些 C++ 数据转换为 UI 侧数据：

```text
mo_ecat::SlaveInfo       -> SlaveTableModel row
mo_ecat::MasterState     -> UI state indicator
mo_ecat log callback     -> LogModel row
mo_ecat::SlaveFeedback   -> Runtime feedback model
```

如果 UI 需要保存配置，建议定义 UI 配置模型，再由 bridge 显式转换为 `EcMasterConfig`。不要让 UI 的配置文件格式直接成为 core 的内部配置格式。

## 7. 状态与命令边界

UI 可以显示主站状态，也可以提供命令按钮，但状态迁移规则由 `mo_ecat_core` 决定。

建议 UI 工作流按照 core 状态机组织：

```text
Uninitialized
  -> InitializeAdapter
AdapterReady
  -> Scan
Scanned
  -> EnterMaintenance
Maintenance
  -> PrepareRun
ReadyToRun
  -> StartOperation
Operational
  -> BackToMaintenance / Stop
Fault / EmergencyStop
  -> Stop
```

UI 可以做按钮可用性控制，但不能把 UI 判断当作最终安全条件。最终是否允许执行某个动作，以 core 返回值和 core 状态机为准。

## 8. 构建边界

当前目标构建方式：

- 根工程使用 CMake。
- UI 使用 Qt6，优先 Qt Widgets。
- `mo_ecat_core` 作为 git submodule 引入。
- UI 工程只链接 `mo_ecat_core` 导出的库或通过 `add_subdirectory` 引入核心库。

根工程后续建议结构：

```text
mo_ecat_pc/
├── CMakeLists.txt
├── app/
│   ├── CMakeLists.txt
│   ├── main.cpp
│   ├── core_bridge/
│   ├── gui/
│   └── app_controller/
├── docs/
├── mo_ecat_core/
└── qt6_demo/
```

`qt6_demo/` 只用于验证本机 Qt6 开发环境，不作为正式 UI 代码。

## 9. Mock 与脱机模式

为降低 UI 开发对 EtherCAT 硬件的依赖，建议在 bridge 层定义 UI 需要的服务接口，例如：

```text
IEcatService
├── RealEcatService  -> MoEcatMaster
└── MockEcatService  -> simulated states/slaves/logs
```

这样 UI 可以在没有从站、没有实时网卡权限的情况下开发和演示。Mock 不应进入 `mo_ecat_core`，除非未来 core 自身需要单元测试用的仿真后端。

## 10. 待确认事项

以下事项会影响后续实现细节，需要 owner 确认：

1. 第一版 UI 是否确定使用 Qt Widgets，而不是 QML。
2. `Service()` 目标周期是 1 ms、500 us，还是只满足 UI 验证阶段的低频调度。
3. 第一版是否必须支持无硬件 Mock 模式。
4. 第一版 UI 功能范围：状态显示、从站列表、日志、SDO 面板、PDO 映射、运行反馈各自优先级。
5. 是否要在 core 中补齐 `on_fault` 和 `on_feedback` 的实际触发逻辑后再接 UI。
