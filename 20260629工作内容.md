## 3. 为什么重构状态机

旧状态机主要使用以下扁平枚举：

```text
kUninitialized
kAdapterReady
kScanned
kMaintenance
kReadyToRun
kOperational
kFault
kEmergencyStop
```

主要问题是：

1. 名称难以区分“主状态”和“动作结果”。
2. `Scanned`、`Maintenance`、`ReadyToRun` 实际上都是准备过程的不同阶段，却被表示成同一层级的主状态。
3. 无法表达“进入中”和“退出中”。
4. UI、CLI、Activity、周期引擎容易各自解释状态，语义会逐渐分裂。

用户认可的新模型是：

```text
五大主状态：初始化态、准备态、运行态、故障态、紧急停止态
迁移阶段：进入中、稳定、退出中
准备态子阶段：网卡就绪、拓扑已发现、PREOP维护、SAFEOP就绪
```

对应公共类型位于：

```text
mo_ecat_core/include/mo_ecat/types.h
```

核心结构为：

```cpp
struct MasterRuntimeState {
    MasterMode mode;
    TransitionStage transition;
    PrepareStage prepare_stage;
};
```

### 4.1 新增分层运行时状态

已新增：

```text

```

旧 `MasterState` 暂时保留，目的是兼容已有公共接口和回调。现阶段不要直接删除。

### 4.2 状态合法性和统一中文显示

实现文件：

```text
mo_ecat_core/src/core/runtime_state.cpp
```

已提供：

```cpp
bool IsValidRuntimeState(const MasterRuntimeState &state);
bool CanRunMaintenanceActivity(const MasterRuntimeState &state);
bool CanRunProcessData(const MasterRuntimeState &state);
bool ShouldCheckPreOpSlaves(const MasterRuntimeState &state);
bool ShouldCheckOperationalSlaves(const MasterRuntimeState &state);
std::string ToDisplayString(const MasterRuntimeState &state);
```

合法性原则：

- 只有准备态可以携带 `PrepareStage`。
- 初始化、运行、故障、紧急停止必须使用 `PrepareStage::kNone`。
- 维护 Activity 只允许在“准备态 / 稳定 / PREOP维护”执行。
- PDO 只允许在“运行态 / 稳定”执行。

### 4.3 进入中和退出中

当前过渡态由以下文件中的 `MoEcatMaster::Impl` 临时维护：

```text
mo_ecat_core/src/core/mo_ecat_master.cpp
```

当前实现使用：

```text
runtime_state_override_
BeginRuntimeTransition()
EndRuntimeTransition()
CurrentRuntimeState()
on_runtime_state_changed
```

生命周期 API 执行前发布进入中或退出中，执行结束后清除覆盖并映射到控制器真实稳定状态。

例如：

```text
StartOperation(): 进入运行态 -> 运行态或故障态
BackToMaintenance(): 退出运行态 -> 准备态 / PREOP维护
Stop(): 退出当前态 -> 初始化态
```

这是过渡实现，不是最终状态所有权设计。下一轮重构应把它下沉到 `EcatController`。

### 4.4 Activity 和周期引擎已经使用新状态

涉及：

```text
mo_ecat_core/src/activity/activity.h
mo_ecat_core/src/activity/activity_scheduler.*
mo_ecat_core/src/activity/*_activity.*
mo_ecat_core/src/cyclic/process_data_engine.*
```

已经完成：

- `Activity::CanStart()` 入参由 `ControllerState` 改成 `MasterRuntimeState`。
- `ActivityScheduler` 通过状态提供器读取运行时状态。
- 所有手动维护 Activity 初期只允许在稳定 PREOP维护态运行。
- `ProcessDataEngine` 通过状态提供器判断是否运行 PDO 和巡检从站。
- `ProcessDataEngine` 不再直接读取 `ControllerState` 决定调度许可。
- `EcatController` 引用仍保留，用于发现异常时请求 Fault。

### 4.5 修复 BackToMaintenance

旧实现中 `ReadyToRun/Operational -> Maintenance` 的真实从站状态回退不完整。

现在 `BackToMaintenance()` 会真实请求从站回到 PREOP，并检查所有从站状态，失败时进入 Fault。

### 4.6 修复 OP 进入握手

首次真机测试时，`prepare` 可以进入 SAFEOP，但 `start` 失败，两个从站都停留在 SAFEOP，AL status code 为 0。

根因是旧 `EcMaster::RequestOperationalState()` 只执行：

```text
写 OP 状态请求 -> 等待状态
```

等待 OP 时没有交换过程数据，而 EtherCAT 从 SAFEOP 进入 OP 通常要求主站持续发送有效 PDO。这会形成“等待 OP 才开始 PDO，但从站等待 PDO 才进入 OP”的顺序问题。

修复位置：

```text
mo_ecat_core/src/ec_master/ec_master.cpp
```

现在流程为：

```text
发送一帧过程数据
写入 OP 请求
在约 2 秒窗口内持续发送/接收过程数据
持续刷新从站状态
全部从站进入 OP 后返回成功
超时时打印最后 WKC、期望 WKC、每个从站状态和 AL status code
```

`CyclicStats` 已增加 `last_wkc`。

另外修复了在已经处于稳定初始化态时再次调用 `Stop()`，不再发布“退出初始化态”。

## 8. 下一步工作：把运行时状态所有权下沉到 EcatController

真机 OP 路径已经通过，下一轮核心重构目标是让 `EcatController` 成为唯一状态源。

### 8.1 当前临时结构

目前存在两层状态认知：

```text
EcatController::state_                 旧 ControllerState，真实稳定状态
MoEcatMaster::Impl::runtime_state_override_  新 MasterRuntimeState 的临时过渡态
```

`ActivityScheduler` 和 `ProcessDataEngine` 通过捕获 `MoEcatMaster::Impl::CurrentRuntimeState()` 的 lambda 获取状态。

这个结构可以工作，但状态所有权分散，不是最终架构。

### 8.2 目标结构

目标应为：

```text
EcatController
  - 持有或生成唯一 MasterRuntimeState
  - 在 Dispatch/动作执行前发布进入中或退出中
  - 动作成功后发布稳定态
  - 动作失败后发布故障态/紧急停止态
  - 对外提供 GetRuntimeState()
  - 提供运行时状态变化通知

MoEcatMaster::Impl
  - 只转发公共 API
  - 将 EcatController 状态通知转发给 on_runtime_state_changed
  - 不再持有 runtime_state_override_

ActivityScheduler / ProcessDataEngine
  - 状态提供器直接读取 EcatController::GetRuntimeState()
```

### 8.3 建议实施顺序

1. 先完成 `back` 真机短路径验证，保留日志。
2. 在 `EcatController` 增加 `GetRuntimeState()`。
3. 把当前 `ControllerState -> MasterRuntimeState` 映射从 `mo_ecat_master.cpp` 移到控制器附近或统一状态模块，确保只有一份映射。
4. 在 `EcatController::Dispatch()` 周围建立运行时状态发布入口。
5. 将每个 `MasterAction` 对应的进入/退出目标集中定义，不要散落在各公开 API 包装函数里。
6. 将 `ActivityScheduler` 和 `ProcessDataEngine` 的状态提供器切到 `controller_.GetRuntimeState()`。
7. 将 `MoEcatMaster::Impl` 中以下内容删除：

   ```text
   runtime_state_override_
   BeginRuntimeTransition()
   EndRuntimeTransition()
   CurrentRuntimeState() 中的覆盖逻辑
   ```

8. 保留 `MoEcatMaster::GetRuntimeState()`，内部直接转发控制器。
9. 保留旧 `MasterState` 和 `on_state_changed` 兼容接口，暂时不要一次性删除。
10. 构建 `mo_ecat_core` 和 `examples/cli`。
11. 使用 CLI 重新验证完整生命周期、back、fault、estop、stop。

### 8.4 重构注意事项

- 不要再次破坏 OP 期间持续交换过程数据的逻辑。
- `transition != kStable` 时，PDO 和维护 Activity 必须保持禁止。
- 失败动作结束后，外部应直接看到故障态，不应短暂回到旧稳定态。
- `Stop()` 在稳定初始化态必须是幂等操作，不发布退出初始化态。
- `BackToMaintenance()` 从运行态必须真实请求所有从站回到 PREOP。
- 内部 `ControllerState` 可以暂时保留，用于映射和渐进迁移，不要在同一个提交中同时删除所有旧状态。
- 当前重点是核心库与 CLI，不继续扩展 Qt UI。
